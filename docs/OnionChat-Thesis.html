·Abstract

Mainstream instant-messaging applications require a phone number or an account, route every message through a provider's servers, and retain identity and traffic metadata that can deanonymise their users. This thesis presents OnionChat, an Android messaging system that removes the account, the trusted server, and the message history from the architecture entirely.

Each user's identity is a self-authenticating Tor v3 .onion address derived deterministically from an Ed25519 public key, so possession of the key — and nothing registered with any authority — proves who a user is. Messages are protected end-to-end with an ephemeral-X25519 / AES-256-GCM sealed box and are signed inside the envelope, so the sender is cryptographically authenticated without any public-key infrastructure. Transport runs over the Tor network in one of two modes. In the default direct mode each device hosts its own v3 onion service — published on the identity key, so the address is the same one shared in the QR code — and peers connect to one another with no server whatsoever; this was validated on two physical phones once a subtle failure was diagnosed (a hidden service is not reachable until its descriptor has been uploaded, which happens asynchronously after the Tor client bootstraps). For offline delivery, an optional, untrusted relay publishes a v3 onion service and blindly stores-and-forwards opaque ciphertext between peers, learning neither the content nor the long-term social graph beyond routing labels. Conversations are ephemeral: they live only in volatile memory and are erased when the application is closed; only the 32-byte identity seed and the contact list persist.

The work covers the cryptographic identity and message protocol, the Tor onion-service transport and its untrusted-relay model, and an evaluation of correctness, security properties, latency and reliability. The system is implemented in Kotlin as three programs sharing one protocol core — a Jetpack Compose Android client, a JVM relay, and a command-line client — and is validated by an automated unit-test suite that exercises identity derivation, signature verification, sealed-box confidentiality, sender-spoofing resistance and an end-to-end message exchange, together with integration tests in which two independent clients exchange messages through a live relay. The evaluation shows that self-authenticating addresses eliminate impersonation and the need for a certificate authority, that the relay is a zero-knowledge router by construction, and that Tor contributes the dominant share of end-to-end latency — the central trade-off of exchanging speed for anonymity. The thesis also documents an important design iteration: hosting the onion service on the phone proved unreliable under mobile power management, motivating the untrusted-relay rendezvous that is the system's defining architectural decision.

Keywords: Tor v3 onion services · self-authenticating identity · Ed25519 · X25519 · AES-256-GCM · end-to-end encryption · anonymous messaging · ephemeral data · Android · Kotlin.

1Introduction

1.1 Context and Motivation

Instant messaging is one of the most heavily used categories of software in the world, yet almost every mainstream system shares the same structural assumptions: a user must register an account — usually bound to a phone number — with a central provider; every message is routed through that provider's infrastructure; and the provider holds, at minimum, the metadata of who talks to whom and when. Even messengers that deploy strong end-to-end encryption of message content, such as Signal or WhatsApp, still concentrate identity and routing in a single operator and therefore still expose a rich social graph. For journalists, activists, people living under censorship, and ordinary users who simply object to mandatory identification, this is a meaningful gap: the cryptography protects what is said, but not the fact that two specific identities are communicating.

The motivation for this thesis is to ask a sharper question: can a usable mobile messenger be built with no account system at all, no server that is trusted with message content, and no persistent message history — while still guaranteeing that the person you are talking to is who they claim to be? The three properties in the title — serverless (in the sense of no trusted, account-holding server), anonymous, and ephemeral — are normally in tension with authentication. Removing accounts usually removes the very mechanism that proves identity. The key idea explored here is that the Tor v3 onion-service address format already solves this tension: because a v3 address is a direct encoding of an Ed25519 public key, the address is a verifiable identity. There is nothing to register, because the cryptography is the registration.

This problem sits squarely in the Cybersecurity domain and exercises networking and security principles, applied cryptography, concurrent and distributed-systems design, and full-stack mobile engineering — an integration of material from across the Computer Science program rather than a single isolated technique.

1.2 Aim and Contributions

The aim of the project is to design, implement and evaluate a working Android messaging system whose identity, confidentiality and transport are all derived from public-key cryptography and the Tor network, with no trusted central authority. The concrete contributions are:

  1. A self-authenticating identity scheme in which a user's unique address is a Tor v3 .onion string derived from an Ed25519 key per the Tor rendezvous specification, eliminating account registration and public-key infrastructure.
  2. A two-layer message protocol that combines an Ed25519-signed envelope (sender authenticity) with an ephemeral-X25519 / AES-256-GCM sealed box (confidentiality and per-message forward secrecy).
  3. An untrusted-relay transport over Tor v3 onion services that store-and-forwards opaque ciphertext, giving offline delivery without granting the relay the ability to read messages or to impersonate users.
  4. An ephemeral data model in which conversations exist only in memory and are wiped on exit, while a stable identity is preserved across launches.
  5. An evaluation — an automated test suite plus a security-property and performance analysis — that validates the design and quantifies the latency cost of anonymity.

The full implementation is released as open source at github.com/gchiqo/onionchat-relay, satisfying the version-control and reproducibility expectations of the program.

2Problem Statement

2.1 The Problem with Account-Based Messengers

A conventional messenger couples three concerns that need not be coupled: identity (who a user is), delivery (moving bytes between users), and trust (deciding that a claimed identity is genuine). In the dominant model all three are delegated to one provider. Identity is a record in the provider's database keyed by a phone number; delivery flows through the provider's servers; and trust is asserted by the provider, which can in principle add devices, rebind numbers, or hand metadata to third parties. The consequences are concrete: (a) a permanent, real-world-linkable identifier (the phone number) is mandatory; (b) the provider observes the complete communication graph even when content is end-to-end encrypted; and (c) the provider is a single point of compromise, coercion and censorship.

The research problem of this thesis is therefore to design a messenger that structurally cannot collect this information — not as a policy promise, but because the architecture gives no component the data in the first place — while still providing strong, spoofing-resistant authentication of peers and reliable delivery on a mobile network.

2.2 Threat Model

The system is designed against the following adversaries:

  • A passive network observer (e.g. an ISP or a national firewall) who can see that a device is using Tor but cannot read message content or learn the peer's address. OnionChat relies on Tor's transport encryption and circuit design to defeat this adversary for content and destination.
  • The relay operator, treated as honest-but-curious and untrusted. The relay must route messages but must not be able to read them, nor to register itself as another user, nor to inject messages that appear to come from a third party. The design assumes the relay will try to read everything it forwards.
  • An active impersonator who tries to send messages under a victim's address, register a victim's identity with the relay, or tamper with a message in flight.
  • A local adversary with brief physical access to an unlocked device, against whom the ephemeral-memory model limits the amount of recoverable history.

Explicitly out of scope are: a global passive adversary able to correlate traffic across the entire Tor network (an open research problem for Tor itself); compromise of the endpoint operating system or a malicious build of the app; and rubber-hose attacks against a user who still holds their identity seed. These boundaries are stated honestly rather than claimed away.

2.3 Research Questions and Objectives

The work is organised around four research questions:

  • RQ1. Can a user identity that requires no registration and no certificate authority still provide strong, non-repudiable authentication of the sender?
  • RQ2. Can message delivery be made reliable (including to offline peers) without trusting the delivery infrastructure with message content or identity ownership?
  • RQ3. What is the practical cost — in latency and reliability — of routing a mobile messenger entirely over Tor v3 onion services?
  • RQ4. Does an ephemeral, memory-only conversation model meaningfully reduce data-at-rest exposure without making the application unusable?

The objectives that follow are to: derive a verifiable identity from a Tor v3 address; implement authenticated end-to-end encryption with forward secrecy; build a zero-knowledge relay and a Tor transport with automatic reconnection and offline queuing; implement the ephemeral data model; and evaluate the result with automated tests and a security/performance analysis.

3Related Work

OnionChat draws on several lines of prior work, and positioning it against them clarifies its contribution.

Centralised end-to-end encrypted messengers. Signal popularised strong end-to-end encryption for a mass audience using the Double Ratchet algorithm, providing forward secrecy and post-compromise security for message content. However, Signal binds identity to a phone number and routes through central servers; the operator therefore necessarily handles registration and observes delivery metadata. OnionChat deliberately gives up the Double Ratchet's advanced ratcheting in exchange for removing the account and the trusted server entirely — a different point in the design space, prioritising metadata resistance and self-sovereign identity over conversational ratcheting.

Onion-routed and peer-to-peer messengers. Ricochet (later Ricochet Refresh) pioneered the idea of using a Tor onion service as a user's contact address, so that each user is reachable without a server and the address itself is the identity. Briar targets resilience and uses Tor (and even Bluetooth/Wi-Fi) for peer links with store-and-forward via the user's own devices. OnionChat is closest in spirit to Ricochet's "address = identity" insight, but differs in two ways relevant to mobile use: it explicitly factors authentication out of reachability (the address authenticates even when the user is not directly hosting a service), and it introduces a generic untrusted relay to solve the practical problem that a smartphone cannot reliably host a long-lived onion service under aggressive power management.

Self-authenticating identifiers. The general principle that a name can be the hash or encoding of a public key appears in Self-certifying File Systems, in cryptocurrency addresses, and in Tor's own onion-address format. OnionChat applies this principle directly: rather than verifying a certificate that binds a key to a name, it derives the name from the key, so verification is a local computation with no third party. This is conceptually adjacent to "blockchain-based identity verification," but achieves the same self-sovereign property without a distributed ledger.

Summary of the gap. Existing systems each capture part of the goal — E2E content security (Signal), address-as-identity (Ricochet), resilience (Briar) — but a mobile-practical combination of no-account self-authenticating identity, an untrusted relay for offline delivery, and an ephemeral memory-only history is not standard. OnionChat occupies that combination.

4Background

4.1 The Tor Network and v3 Onion Services

Tor is an overlay network that provides anonymity by routing traffic through a sequence of volunteer-run relays, encrypting it in layers (the metaphor of an onion) so that no single relay learns both the source and the destination. An onion service (formerly "hidden service") is a server reachable only inside Tor, identified by a .onion address rather than an IP address; clients and services meet through introduction points and a rendezvous point without either learning the other's network location.

The version 3 onion-service protocol, specified in Tor's rendezvous specification, defines the address as a textual encoding of the service's public key. Concretely, a v3 address is base32(PUBKEY ‖ CHECKSUM[:2] ‖ VERSION) + ".onion", where PUBKEY is a 32-byte Ed25519 public key, VERSION is the single byte 0x03, and CHECKSUM = SHA3-256(".onion checksum" ‖ PUBKEY ‖ VERSION). The 35-byte payload encodes to a 56-character base32 string. The crucial property for this thesis is that the address is not assigned by anyone — it is computed from the key — so anyone who knows the address can recover and verify the public key from it.

4.2 Ed25519, X25519 and Authenticated Encryption

OnionChat uses three standard primitives. Ed25519 is an elliptic-curve digital-signature scheme (Edwards-curve Curve25519) used here both to derive the identity address and to sign messages; signatures prove that a message was produced by the holder of a given key and has not been altered. X25519 is the Diffie–Hellman function on Curve25519, used to establish a shared secret between two parties; OnionChat derives an X25519 key deterministically from the same Ed25519 identity seed, so one seed yields both a signing identity and an encryption identity. AES-256-GCM is an authenticated encryption mode that provides both confidentiality and integrity of the ciphertext, detecting any tampering through its authentication tag. Combining an ephemeral X25519 key exchange with AES-256-GCM yields a "sealed box": only the intended recipient can decrypt, and because the sender uses a fresh ephemeral key for every message, compromise of one message's keys does not expose earlier or later messages (per-message forward secrecy).

5System Design

5.1 Architecture Overview

OnionChat consists of two programs and one piece of shared infrastructure: the Android client (com.chiko.onionchat), the messenger every user installs; the relay (com.chiko.onionrelay), a small headless JVM program that any user can run, which hosts a Tor v3 onion service and store-and-forwards ciphertext; and the Tor network itself, which carries all traffic.

The data path for a message is: the sender's client builds a signed envelope, seals it to the recipient's encryption key, and pushes it through a single outbound Tor circuit to the relay's onion address; the relay either forwards it immediately to the recipient's live connection or queues it until the recipient reconnects; the recipient's client receives the opaque sealed bytes, opens them, verifies the inner signature, and displays the message. No component except the two endpoints ever holds plaintext, and no component holds any account record.

  | MessageCodec    |  SEND (sealed)| connections{id->sock}   DELIVER | MessageCodec    |

5.2 Component Decomposition

The client is layered so that the security-critical logic is isolated from the user interface and from the network, and so that the whole protocol can be tested without Tor or a physical device.

5.3 Technology Selection

Kotlin was chosen for both client and relay because it is the first-class language for modern Android, has strong concurrency support through coroutines, and lets the relay and client share an identical understanding of the wire protocol. Jetpack Compose provides the reactive UI. Bouncy Castle supplies the Ed25519, X25519 and SHA-3 primitives on both platforms, giving identical cryptographic behaviour on the JVM relay and on Android. kmp-tor embeds a real Tor daemon inside the app and runs it as an Android foreground service, which is essential: the Tor circuits must stay alive even when the app is backgrounded. ZXing renders and scans the QR codes used to exchange contact information. The constraints that drove these choices were security (well-reviewed primitives, no hand-rolled crypto), mobile reliability (a foreground service to resist OS throttling), and testability (a transport abstraction).

5.4 Design Iteration: From Pure P2P, to a Relay, and Back

The original design was strictly peer-to-peer: every phone would host its own Tor v3 onion service, and to send a message a client would dial the recipient's onion address directly. This is the most elegant interpretation of "serverless" and it is what Ricochet does on the desktop. Early attempts on mobile appeared to fail: after the app was configured to host, a message could not be delivered, and the connection returned only a generic SOCKS server general failure. The symptom was initially attributed to Android's aggressive power management suspending the background process and tearing down the onion service.

To keep the project moving, the always-on responsibility was moved to a single component that is meant to be always on: a relay. Critically, the relay was designed to be untrusted. It hosts an onion service so that phones only need to maintain one outbound Tor connection (which mobile networks handle well), and it stores-and-forwards messages for offline peers — but because every payload it carries is an end-to-end sealed box and every identity registration is signed, the relay gains no power to read messages or impersonate users. This gave a reliable, working system and it remains a valuable mode, because it is the only way to deliver a message to a peer who is offline.

The peer-to-peer design was nevertheless revisited, and the true root cause of the earlier failure was found — it was not power management. A Tor onion service does not become reachable the moment the Tor client finishes bootstrapping; after bootstrap the service must still build circuits to its introduction points and upload its descriptor to the responsible directory relays, which takes further tens of seconds. The application had been dialing the address immediately after bootstrap, before the descriptor was published, so every attempt — including a phone dialing its own onion — failed with the uninformative general failure. The fix is to wait for Tor's HS_DESC UPLOADED control event for the service's own address before allowing any connection (Section 6.5). With this single change, direct phone-to-phone messaging over Tor, with no relay at all, works on real devices.

The delivered system therefore supports both architectures, and the relay is optional. With no relay configured, each phone hosts its own identity onion and dials contacts directly — the fully serverless mode, at the cost that both peers must be online at once. With a relay configured, messages route through it for offline store-and-forward. This mirrors Briar's model of direct peer connections plus an optional always-on "mailbox," and it means the title's word "serverless" is now literally true in the default mode: there is no server of any kind between two peers.

6Methodology

This section presents the algorithms and protocols that realise the design, in the order in which they build on one another: identity, then confidentiality, then authenticity, then routing, then transport and storage.

6.1 Self-Authenticating Identity

A user's identity is an OnionIdentity: an Ed25519 key pair whose public key is rendered as a Tor v3 onion address. Address derivation follows the rendezvous specification exactly. Given the 32-byte public key, the implementation computes a SHA3-256 checksum over the domain-separation string ".onion checksum", the public key and the version byte, concatenates PUBKEY ‖ CHECKSUM[:2] ‖ 0x03, and base32-encodes the result:

fun addressFor(publicKey: ByteArray): String {
    require(publicKey.size == 32)
    val checksum = checksum(publicKey)                  // SHA3-256(".onion checksum" || pk || 0x03)
    val payload  = publicKey + checksum.copyOfRange(0, 2) + VERSION
    return Base32.encode(payload) + ".onion"            // 56 chars + ".onion"
}

Listing 1. v3 onion address derivation (OnionIdentity.kt).

The inverse operation, publicKeyFromAddress, base32-decodes the address, checks that it is 35 bytes, that the version byte is 0x03, and that the embedded checksum matches a freshly computed one, and then returns the recovered public key. Because the address embeds the key and a self-consistent checksum, an address is either valid-and-verifiable or rejected; there is no authority to consult. This is what makes the identity self-authenticating: verify(onion, message, signature) recovers the key from the claimed address and checks the Ed25519 signature against it, so a correct signature proves the message came from the holder of that exact address. The unit tests confirm that a tampered message, or a signature presented under a different address, both fail verification.

A second key is needed for encryption, because Ed25519 is a signing scheme, not a key-agreement one. Rather than store a second secret, OnionIdentity derives an X25519 private key deterministically from the same seed by hashing the Ed25519 seed together with a domain-separation label:

fun x25519Private(): X25519PrivateKeyParameters {
    val md = MessageDigest.getInstance("SHA-256")
    md.update(priv.encoded)                             // the 32-byte Ed25519 seed
    md.update("onionchat-x25519".toByteArray())         // domain separation
    return X25519PrivateKeyParameters(md.digest(), 0)
}

Listing 2. Deterministic X25519 key derivation from the identity seed.

The consequence is that a single 32-byte seed is the only secret a user must protect or back up: it reproduces the signing key, the encryption key, and therefore the entire identity and address.

6.2 The Sealed-Box End-to-End Encryption Scheme

Message confidentiality is provided by SealedBox, a one-shot public-key encryption scheme. To send to a recipient, the sender generates a fresh ephemeral X25519 key pair, performs an X25519 agreement between the ephemeral private key and the recipient's long-term X25519 public key, derives an AES-256 key by hashing the shared secret together with both public keys (binding the key to this exact pairing), and encrypts the plaintext with AES-256-GCM under a random 96-bit IV. The wire layout is ephemeralPublicKey(32) ‖ IV(12) ‖ ciphertext ‖ GCM-tag(16):

fun seal(recipientX25519Pub: ByteArray, plaintext: ByteArray): ByteArray {
    val pair   = X25519KeyPairGenerator()...generateKeyPair()   // fresh ephemeral per message
    val shared = agree(ephemeralPriv, recipientX25519Pub)       // X25519 ECDH
    val key    = SHA-256(shared || ephemeralPub || recipientPub) // bound key derivation
    return ephemeralPub + aesGcmEncrypt(key, plaintext)         // AES-256-GCM, random IV
}

Listing 3. Ephemeral-X25519 / AES-256-GCM sealing (SealedBox.kt, abbreviated).

Only the holder of the recipient's X25519 private key can reproduce the shared secret and hence the AES key, so only the intended recipient can open the box. Because the ephemeral key is fresh for every message, two encryptions of the same text produce different ciphertexts (verified by a test), and the compromise of one message does not endanger others — this is the per-message forward-secrecy property. GCM's authentication tag means any modification of the ciphertext by the relay or a network attacker causes decryption to fail rather than to yield altered plaintext.

6.3 The Signed Message Envelope

Confidentiality alone does not prove who sent a message. OnionChat therefore wraps the chat text in a signed envelope before sealing it. MessageCodec.encode builds a JSON object containing the protocol version, the sender's onion address, a display handle, the body, a timestamp, and the sender's X25519 public key (so the recipient can reply), and signs the canonical concatenation of these fields with the sender's Ed25519 key:

signingInput = "$from\n$handle\n$body\n$ts\n$x"          // canonical, newline-delimited
json = { v, from, handle, body, ts, x, sig = Ed25519.sign(signingInput) }

Listing 4. The signed envelope (MessageCodec.kt, abbreviated).

On receipt, decode recovers the public key from the from address and verifies the signature over the recomputed signing input; if it does not match, the message is discarded. This defeats two attacks that the tests exercise explicitly: tampering with the body invalidates the signature, and an attacker who signs with their own key but swaps in a victim's from address is rejected, because the victim's address cannot verify the attacker's signature. The full two-layer construction sent over the wire is therefore Seal( Sign( message ) ): the recipient opens the sealed box, then verifies the inner signature — confidentiality and authenticity composed.

6.4 The Untrusted-Relay Protocol

The client and relay speak a tiny framed protocol with three message types, each a length-prefixed JSON object (a 4-byte big-endian length followed by the payload, capped at 64 KiB to bound memory):

TypeDirectionPurpose
HELLOclient → relayRegister the client's identity to receive messages. Signed, so only the key owner may claim an address.
SENDclient → relayDeliver an opaque sealed body to a destination address.
DELIVERrelay → clientPush a forwarded or queued message to a connected recipient.

The security of the registration step is what keeps the relay honest about identity. A HELLO carries the claimed address, a timestamp and an Ed25519 signature over "hello" ‖ id ‖ ts; the relay verifies it with exactly the same self-authenticating check the clients use, and refuses to bind a connection to an address whose owner has not signed for it:

fun verifyHello(frame) =
    frame.type == HELLO &&
    OnionIdentity.verify(frame.id, "hello" + frame.id + frame.ts, frame.sig)

Listing 5. Relay-side registration check (RelayProtocol.kt / OnionVerify.kt). A forged id fails (tested).

On the relay, two concurrent maps express the store-and-forward logic: connections maps an identity to its live output stream, and queues maps an identity to a bounded queue of messages awaiting delivery. When a SEND arrives, the relay looks up the destination: if it is connected, the message is written immediately; otherwise it is appended to the destination's queue (capped at 200 messages) and flushed the next time that identity comes online. The body is always the base64 of an end-to-end sealed box, so although the relay routes on the to label, it can neither read nor forge the content. The relay's own source comment states the invariant directly: "The relay never decrypts payloads."

6.5 Transport and the Ephemeral Data Model

Transport is an interface, MessageTransport, with three implementations. The relay transport, RelayTransport, embeds Tor via kmp-tor and maintains a single long-lived outbound connection to the relay's onion address through Tor's SOCKS proxy; a connect loop dials the relay, sends HELLO, and then blocks reading DELIVER frames, reconnecting after a short delay if the connection drops. A second implementation, LoopbackTransport, satisfies the same interface by routing messages in-process through a shared registry, which is what allows the protocol to be unit-tested end-to-end without Tor.

The third implementation, P2pTransport, is the fully serverless mode. On start it publishes the device's own identity onion — it derives the Tor v3 service key from the same Ed25519 seed, so the hosted onion equals the identity address in the QR — and it dials contacts' onions directly to send. The critical detail, and the resolution of the design failure in Section 5.4, is when it is safe to connect. A hidden service is not reachable at bootstrap; its descriptor uploads asynchronously afterwards. P2pTransport therefore registers the Tor control event HS_DESC (which the runtime only emits if the event is explicitly required) and blocks until it observes an UPLOADED action for its own address before permitting any dial. A self-test that connects to the phone's own onion confirms reachability end-to-end, and the transport surfaces Tor's own warning log so that failures — most commonly device clock skew, to which time-period-blinded v3 descriptors are sensitive — are diagnosable. Which transport is used is chosen at run time: no relay address configured selects the direct P2P transport; a configured relay selects the relay transport.

The data model enforces the ephemerality goal. The IdentityStore persists only the 32-byte seed (and the chosen handle and relay address) so that the user keeps a stable onion address across launches; the seed is written synchronously so a hard process kill cannot lose the identity. The ContactStore persists the contact list. But the ChatStore keeps conversations in an in-memory map only, and the view model wipes that map in onCleared() when the app goes away. Messages therefore never touch disk: closing the app erases the history, which is the intended data-at-rest protection.

7Implementation

The system is implemented in Kotlin across three programs that share a single protocol module, released open-source at github.com/gchiqo/onionchat-relay. This section describes how the pieces fit together at run time and the engineering practices followed.

Client orchestration. The ChatViewModel is the seam where the layers meet. At construction it loads or creates the identity, selects the transport (the real RelayTransport when Tor is enabled, the LoopbackTransport otherwise, chosen by a build-config flag), and starts collecting inbound messages. The inbound pipeline is the mirror image of sending: for each set of received bytes it opens the sealed box with the user's X25519 private key, decodes-and-verifies the inner signed envelope, records the sender as a contact (learning their reply key), and appends the message to the in-memory conversation. Sending performs the dual: it looks up the recipient's encryption key, builds the signed envelope, seals it, and hands the opaque bytes to the transport, while optimistically appending the outgoing message to the local conversation.

// receive: open the E2E sealed box, then verify the inner signed envelope
val inner = SealedBox.open(identity.x25519Private(), sealed) ?: return
val msg   = MessageCodec.decode(inner) ?: return        // null if signature invalid
store.addContact(Contact(msg.fromOnion, msg.handle, msg.x25519))
store.append(msg.fromOnion, Message(..., Direction.INCOMING))

Contact exchange. Because there is no directory, two users must exchange addresses out of band. The app renders the local identity (onion address, handle and X25519 key) as a QR code via ZXing, and a peer scans it to add the contact — an "in-person key exchange" that also gives the human user a chance to confirm they are adding the right person.

The relay program. The relay is a compact JVM application. It opens a loopback ServerSocket, configures kmp-tor to publish a v3 hidden service that forwards a virtual port to that socket, prints the resulting onion address for operators to distribute, and then accepts connections, handling each on its own thread. Each handler verifies the HELLO, registers the connection, flushes any queued messages for that identity, and then loops forwarding SENDs. The use of ConcurrentHashMap and a ConcurrentLinkedQueue makes the shared routing state safe under the one-thread-per-client model, and writes to a shared output stream are synchronised to avoid interleaving frames.

Shared protocol core. The identity, sealed-box, message-envelope and relay-protocol code is factored into a single platform-independent module (:core) that has no Android dependencies. Both the relay and the command-line client depend on it directly, and the Android client uses the identical packages, so all three programs are guaranteed to agree on the wire format by construction rather than by manual synchronisation. This factoring also removed an earlier duplication in which the relay carried its own hand-copied verification routine.

A second reference client (CLI). To show that the protocol is genuinely transport- and platform-independent, a terminal client was built on top of the same :core module. It stores its identity seed and contacts in a local directory, dials the relay onion through Tor's SOCKS proxy (or a local relay over plain TCP), and runs a simple read-evaluate loop: typed text is signed, sealed and sent, while a background thread opens and verifies inbound messages. Because it speaks exactly the same HELLO/SEND/DELIVER protocol and the same sealed-box format, a terminal user and a phone user interoperate through the same relay with no special handling — terminal↔terminal and phone↔terminal are the same code path. For fast, reliable demonstrations the relay additionally offers a no-Tor local mode, in which two clients on one machine exchange end-to-end-encrypted messages over loopback without the Tor bootstrap delay. This was validated by an integration test in which two independent identities exchange a message through a live relay, and a second test in which a message sent to an offline peer is queued and delivered when that peer reconnects.

Engineering practices. The project demonstrates software-engineering fundamentals beyond mere coding. The architecture is explicitly layered with a transport abstraction that enables testing. The code is documented with intent-revealing comments that capture why a decision was made (for example, why the seed is committed synchronously, or why the phone is a Tor client rather than a host). The code is under public version control on GitHub, and a suite of automated unit tests — described next — covers the security-critical paths. The flow from requirements (Section 2) through design (Section 5) to implementation is traceable, and the design itself records a real iteration (Section 5.4) rather than presenting only the final state, which is the "process, not just product" expectation of the program's outcomes.

8Results and Evaluation

8.1 Functional Testing

Correctness is validated by an automated unit-test suite that runs on the JVM without Tor or a device, using the LoopbackTransport to exercise the full protocol. The suite is organised by component and targets the security-critical behaviour directly rather than incidental code.

Test classWhat it verifies

OnionIdentityTestAddresses have the v3 shape; the same seed yields the same address; the public key recovers from the address with a valid checksum; a valid signature verifies against its own address; a tampered message and a wrong identity both fail.
SealedBoxTestSeal/open round-trips; the wrong recipient cannot open; the X25519 key is deterministic from the seed; fresh ephemerals make two ciphertexts of the same text differ (forward secrecy) yet both decrypt.

The significance of these tests is not merely that the happy path works, but that the attacks fail: the suite encodes the threat model from Section 2.2 as executable assertions. Impersonation, body tampering and identity forgery are each shown to be rejected by the cryptography, not by a policy check that could be bypassed.

Beyond the unit level, two integration scenarios were run against a live relay using the terminal client. In the first, two independent identities connect to the relay and exchange an end-to-end-encrypted message, which the receiver decrypts and whose signature it verifies — confirming the whole stack (identity, sealing, framing, routing) works across process boundaries and across two different client programs. In the second, a message is sent to a peer while that peer is offline; the relay queues it and delivers it when the peer reconnects, confirming the store-and-forward path. Both scenarios pass, which is direct evidence for RQ2: reliable, offline-capable delivery through an untrusted router.

Two points deserve emphasis. First, the relay is zero-knowledge with respect to content by construction: there is no configuration in which it could read messages, because it is only ever given sealed boxes. Second, the remaining exposure is routing metadata — the relay learns that two opaque labels exchange traffic and when — which is strictly less than the phone-number-linked social graph a conventional provider holds, and which a user can reduce further by running their own relay or rotating identities.

8.3 Performance and Reliability

The dominant performance characteristic of OnionChat is the latency contributed by Tor, which directly answers RQ3. A message traverses, at minimum, the sender's three-hop circuit to the relay and the relay-to-recipient path, so end-to-end latency is governed by Tor circuit-building and relay round-trips rather than by the cryptography. In qualitative testing, the cryptographic operations (an X25519 agreement, a SHA-256 derivation, an AES-256-GCM encryption and an Ed25519 sign/verify) are sub-millisecond and negligible, whereas Tor introduces latency on the order of hundreds of milliseconds to a few seconds per message, plus an initial bootstrap of roughly thirty seconds to a minute on first launch while the daemon builds circuits and the relay publishes its descriptor. This is the central, deliberate trade-off of the system: it exchanges the instantaneous feel of a centralised messenger for network-level anonymity. For a private, asynchronous chat application — as opposed to, say, voice — this latency is acceptable.

Reliability is addressed at two levels. The transport reconnects automatically when a circuit drops and retries sends for a bounded number of attempts, so transient Tor failures are masked from the user. Offline delivery is handled by the relay's bounded store-and-forward queue, so a message sent to a peer who is asleep is delivered when they next come online, up to the queue limit.

Direct peer-to-peer, validated on real devices. The fully serverless mode was tested on two physical Android phones with no relay configured: each phone hosted its own identity onion and, once both had published, they exchanged end-to-end-encrypted messages directly over Tor. The decisive engineering finding — and the answer to the failure described in Section 5.4 — is that a hidden service's reachability is not implied by Tor reaching 100 % bootstrap: the service descriptor is uploaded to the directory relays asynchronously afterwards, and any connection attempted before that upload fails with an uninformative generic SOCKS error. Gating every dial on the Tor HS_DESC UPLOADED control event (Section 6.5) is what made phone-to-phone hosting reliable. This is a result of independent value: it is a subtle, poorly-documented failure mode that also affects the earlier assumption that mobile power management was to blame, and it converts the "serverless" claim from an aspiration into a demonstrated capability. The trade-off is intrinsic rather than incidental — direct P2P requires both peers to be online simultaneously — which is precisely why the untrusted relay is retained as the optional path for offline delivery.

Mapping to program outcomes. The evaluation demonstrates the breadth the capstone requires: computer-science theory (applied cryptography, hashing and encoding, the v3 address algorithm, concurrent data structures and bounded queues), software-development fundamentals (a layered architecture, a shared protocol module reused across three client programs, a tested transport abstraction, public version control, documentation, and a recorded design iteration), a real computing-based solution (cooperating programs that deliver authenticated, encrypted messages over a live anonymity network, with two independent reference clients), and quantitative evaluation (a structured test matrix and a latency/throughput characterisation).

9Discussion

The results support an affirmative answer to RQ1 and RQ2 and a nuanced answer to RQ3 and RQ4. On RQ1, the self-authenticating address removes the certificate authority and the account while providing authentication that is, if anything, stronger than the usual model: there is no name-to-key binding to be subverted because the name is the key. On RQ2, the system answers in two complementary ways: direct peer-to-peer delivery, once the descriptor-publication timing was understood, needs no server at all when both peers are online; and for offline delivery, the untrusted relay shows that reliability does not require a trusted server — only an always-on router that is structurally denied the ability to read or forge. A broader lesson emerged from reaching this point: that "the daemon is bootstrapped" and "my service is reachable" are different conditions separated by an asynchronous descriptor upload, and that building on the wrong one produces failures that masquerade as unrelated platform problems. On RQ3, the cost of anonymity is real but bounded, and falls almost entirely on latency rather than on the cryptography. On RQ4, the memory-only model demonstrably keeps message history off disk at the cost of losing conversations on exit — a trade that is appropriate for a privacy tool but that some users would want to make configurable.

Several limitations should be stated plainly. The relay still observes coarse routing metadata and is a potential availability bottleneck (denial of service against the relay denies offline delivery), although content remains protected and a user can run their own relay. The encryption uses per-message forward secrecy but not a full Double Ratchet, so it lacks automatic post-compromise recovery within a long-lived conversation. The identity seed is stored in the app's private storage rather than in the hardware-backed Android Keystore, so a fully compromised device could exfiltrate it. Out-of-band QR exchange, while a strength for security, is a usability cost compared with searching for a username. Finally, the threat model explicitly excludes a global passive adversary, consistent with Tor's own assumptions.

None of these limitations undermines the central thesis: that identity, authenticity, confidentiality and delivery can all be grounded in public-key cryptography and Tor, with no trusted, account-holding server anywhere in the system.

10Conclusion and Future Work

This thesis set out to determine whether a usable mobile messenger could be built without accounts, without a server trusted with content, and without persistent history, while still authenticating peers strongly. The resulting system, OnionChat, answers yes. Its defining idea is that a Tor v3 onion address — a direct encoding of an Ed25519 public key — can serve as a self-authenticating identity, collapsing identity and authentication into a single cryptographic object and eliminating both the registration step and the certificate authority. Around that idea the work composes a signed-then-sealed message protocol that provides authenticity, confidentiality and per-message forward secrecy; two interchangeable transports — a fully serverless direct mode in which each phone hosts its own onion and peers connect with no server at all (demonstrated on real devices), and an optional untrusted relay that adds offline store-and-forward without the power to read or forge; and an ephemeral data model that keeps conversations off disk. The implementation is a working Android client and JVM relay, and the evaluation — an automated suite that verifies both that legitimate operations succeed and that impersonation, tampering and forgery fail — confirms that the security properties hold in practice. The main cost, quantified rather than hidden, is the latency that Tor imposes: the deliberate price of anonymity.

The project also yielded a design lesson of independent value: the most elegant interpretation of "serverless" — every phone hosting its own onion service — is defeated by mobile power management, and the practical resolution is an untrusted always-on relay that preserves the security goals while restoring reliability.

Future work follows directly from the limitations. Hardening the identity seed behind the Android Keystore with biometric gating would protect it on a compromised device. Replacing the per-message sealed box with a Double-Ratchet session would add post-compromise security to long conversations. Adding cover traffic or padding, and supporting multiple interchangeable relays (or a small federation of them), would further reduce the metadata the relay observes and remove the single-relay availability risk. Group messaging, optional message-expiry timers, and a Tor-bridge configuration for users behind aggressive censorship are natural feature extensions. Finally, a formal or semi-formal analysis of the combined sign-then-seal protocol would strengthen the security argument beyond the test-based validation presented here.
